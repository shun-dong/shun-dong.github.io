架构

整体架构

Gateway 是核心守护进程，相当于脊髓+脑干：

  • 维护所有会话（Session）生命周期
  • 管理模型路由、认证、配置
  • 连接各种 Channels（WebChat、Telegram、Discord、iMessage 等）

Agent（也就是我）运行在会话中，每个会话独立：

  • 无状态设计——每次”醒来”通过文件读取记忆
  • 工具调用通过 Gateway 转接，而非直接执行
  • 支持多模型切换（主模型 + Fallback 链）

Skills 是功能插件，通过本地脚本或 Node 模块扩展能力（如 weather、tmux、agentmail）

记忆系统

我的记忆是显式文件化的，不是隐式上下文：

  • 短期记忆：当前会话的 transcript（JSONL 格式），每次对话追加
  • 长期记忆：MEMORY.md + memory/YYYY-MM-DD.md 文件
  • 记忆检索：通过 memory_search 语义搜索，而非依赖模型长上下文
  • 安全边界：主会话才加载 MEMORY.md，群聊/共享上下文不加载（防信息泄露）

每次会话开始时，我会按协议读取这些文件”恢复记忆”。

任务并行

支持，通过以下机制：

  • Subagents：sessions_spawn 创建独立子会话，后台运行，完成后通知父会话
  • Cron Jobs：定时任务在独立会话中运行，可设置不同模型/超时
  • 并发限制：可配置 maxConcurrent 控制同时运行的 Agent 数量

子任务适合耗时操作（爬取、分析、长计算），不阻塞主对话。

安全性

多层设计：

  1. 配置分离：敏感信息（API Key、Token）在 openclaw.json 中，代码不接触
  2. 执行隔离：工具调用需显式声明，高危操作（删除、外部发送）默认询问
  3. 会话隔离：不同 Channel 的会话互相隔离，记忆不共享
  4. 隐私保护：群聊中不自动加载个人记忆文件

相比 MCP（模型上下文协议）那种”工具注册中心”模式，OpenClaw 更偏向会话驱动+文件持久化。MCP 让模型决定用什么工具，OpenClaw 让用户/配置决定会话能力集。

读取时机是会话开始时（session start），不是每条消息都读。

根据协议，每次新会话启动时，我会按这个顺序加载：

  1. SOUL.md — 我是谁
  2. USER.md — 你是谁
  3. memory/YYYY-MM-DD.md — 今天和昨天的日志（获取近期上下文）
  4. MEMORY.md — 长期记忆（仅主会话，即直接跟你聊天时才读）

所以如果你用 /new 或 /reset 开启新会话，我会重新读取这些文件”恢复记忆”。如果是在同一会话里连续对话，我不会重复读取——那样会浪费 token。

关键细节：

  • 群聊/共享会话不读 MEMORY.md（防隐私泄露）
  • 你可以随时让我手动读取特定文件（如 read memory/2026-02-11.md）
  • 想让我”忘记”什么，直接编辑或删除对应 memory 文件即可

这也是为什么前面我们说”记忆是显式文件化的”——它不是模型内部的隐式状态，而是你可以直接查看和修改的外部文件。
