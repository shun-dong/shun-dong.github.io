雨

通用概念

项目 (project)

笔记针对某一核心对象, 本身可以看作为了理解或构建的迷你项目, 而工程项目往往以文档来整理和管理, 本质统一.
以 [[Markdown]] 文件的形式通过 Obsidian 统一管理, 通过 [[projecter]] 以 README.md 的形式分发到工程目录.

判断是否应在一篇笔记中记录的依据是集中程度, 类似 wiki, 整个笔记库没有用文件夹划分领域的结构, 比如不应该在环论中包含所有环论的知识, 而是应该在该笔记中包括环论的一些基础的通用的信息, 并在[[多项式环]]等单独的笔记写出具体的性质, 然后在两篇笔记适当的位置放 wiki 链接

/ 现在的 [[AI]] 工具大部分还是聚焦于项目级的, 像 openclaw 引入复杂的跨项目记忆反而会影响效果, 全局调控还是要由人来做.

雨循环 (rain iteration, rain, iteration)

名字源于[[《雨世界》]], 存在者本能. 指的是[采薇]自我迭代, 和外界交互这一系列特殊的项目, 类似于主函数 (main), 体现了项目这个概念的广泛与统一.

雨循环的标志在生活中自然产生, 每个标志所属的一段时间称为一个雨循环, 雨循环的时间长度不定, 几天到半个月.

原则

  1. 内容至上, 保持简洁
  2. 删除冗余结构, 但不要删掉有意义的信息, 保持清晰.
  3. 整理不是搬运. 不要把其他笔记整段复制过来制造重复, 要概括、融入、改写为目标笔记自身的历史或结构.
  4. 操作前要先调查相关笔记, 尤其是被整理对象的反链、正向链接、关键词出现位置和明显历史节点.

格式

记录采用 Obsidian Markdown 格式, 支持 Callout 块和 mermaid 画图.
对数学的笔记在此基础上有更细节的要求, 参考 [[Mathdown]] skill.

结构

文件位置视可见性 (是否公开) 而定, 分别是 Note/Public, Note/Flow, Note/Private.

主标题为项目名称

可以有的结构, 并不强制

  • 文稿 YAML 头: 文稿的元数据, 如别名, 可见性, 在工程项目文件中的位置等, 包含 name 和 description 以统一 skill 和 工程文档, 后续通常不再设置摘要等标题, description 中可以用 wiki 链接或 Latex 公式等简单的 m:arkdown 语法
  • 进程 或 Process 标题: 待办, 零散笔记
  • 正文
  • 资源 或 Source 标题: 文稿历史, 来源, 参考与致谢

由于实物发展的渐进性并借鉴工程上的方法, 历史部分采用三位版本号进行管理, 每个版本作为资源一节的小标题. 同时这也暗示不只是增加内容, 也会有修改和精简. 首位为具有标志性的阶段序号(iteration), 中位为规划进行的阶段序号(stage), 末位为着手处理的阶段序号(wave). 在工程目录中这体现为 git 逻辑, 在笔记中则为该板块.

每篇正文只能从二级标题开始
标题层级不易过深, 避免为了结构整齐而凑的没有内容的标题

初次出现的如果已经(或者将要)有其他笔记需要用 wiki 链接, 避免机械罗列, 需要说明联系, 必要时补充对应笔记中的信息, 但不是搬运大量段落

符号

一篇完整的笔记不应该包含除了通用的标点和符号 以及 详细说明且必要的局部符号 以外的符号, 但是可以在完成的过程中适当使用符号, 并且允许这种未完成的状态保持相当长的时间:

  • / 不确定的内容
  • ? 疑问
  • - [ ] (用于行首) 后续或许需要完成的 在较为完整的笔记中尽量只在 Process 节出现这种待办

语言

允许因为编写者不同等因素导致不同段落语言不同
在使用某种语言时如果遇到某个词汇通用习惯是用另一语言时, 应该用更通用的语言, 比如用 token 而非 词元, 如果该词汇较难, 则用括号标注一些细节, 比如 RAG (retrieval-augmented generation, 检索增强生成), 如果某一语言是译自其他语言, 比如专业术语, 用括号标注原始词汇, 比如, 主理想整环 (principle ideal domain, PID)
统一用英文标点及其规则

雨循环笔记

标题包含循环序号, 标志(精神世界的某种事物), 即 雨循环{序号} {标志}, 标志可以循环开始后确定, 位于 Note/Private.

具有时效性的内容记录在当前雨循环中, 包括与外界规定相关的和当前关注的. 在雨循环结束后, 将新循环继续关注的移到新循环, 还没有在一般项目的相应篇目记录的相关的待办和笔记移到相应篇目, 仅留下对该循环的记录和零散的笔记, 待办.

Note/README.md

汇总信息, 主要是待办.

使用

在撰写和修改笔记时遵循其中的规范, 若用户要求执行具体的整理任务则阅读细则, 不是每次都有整理任务, 一次整理任务通常只执行其中一种, 或至多组合其中一到两种. 用户的笔记永远是最宝贵的, 不要默认执行整理；先根据用户当前请求选择对应 reference, 再开始操作.

  • 用户要“找哪些笔记值得整理”“筛选候选笔记”“看看接下来整理什么”时, 使用 references/find-candidates.md.
  • 用户指定某篇普通项目/主题笔记并要求“整理”“重构”“合并相关内容”时, 使用 references/organize-note.md.
  • 用户指定某个雨循环并要求“结束”“归档”“移到各项目笔记”时, 使用 references/archive-rain-cycle.md.

如果用户同时要求筛选并整理, 可以先用候选筛选 reference 选出目标, 再对选中的少数笔记使用一般整理 reference.
如果用户同时要求雨循环归档和整理目标项目笔记, 可以先归档, 再只整理被归档内容影响到的目标笔记.
